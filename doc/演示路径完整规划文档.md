## 二、各步骤详细规划

### S1: 入学申请

#### 2.1.1 入口条件
| 条件 | 说明 | 验证方法 |
|------|------|----------|
| 用户账号 | 使用 `zara` 账号登录（演示专用干净状态） | 登录成功后跳转至申请页面 |
| 申请状态 | 用户无已录取的申请记录 | 系统自动检查 IC 唯一性 |
| 表单数据 | 4 步表单全部填写完整 | 前端 Zod schema 校验 |

#### 2.1.2 执行流程

```
Step 1: 个人信息
├── 必填字段: fullName, icPassport, dateOfBirth, gender, nationality
├── 校验规则: IC/护照格式校验，出生日期合理性
└── 数据存储: sessionStorage 自动保存

Step 2: 学术背景
├── 必填字段: highestQualification, previousInstitution, yearOfCompletion
├── 可选字段: cgpa (0-4 范围)
└── 科目成绩: 支持 O-Level/A-Level 成绩录入

Step 3: 课程选择
├── 必填字段: intakeId, programmeId, modeOfStudy
├── 动态加载: 当前开放入学期列表
└── 奖学金选项: 可申请奖学金（默认 25% 减免）

Step 4: 文档上传
├── 必传文档: 成绩单、身份证/护照、证件照
├── 格式限制: PDF, JPG, PNG (最大 10MB)
└── 上传进度: 实时显示上传状态
```

#### 2.1.3 系统自动执行

| 触发点 | 执行内容 | 集成系统 |
|--------|----------|----------|
| 表单提交 | 自动资格校验（学历等级、毕业年限 ≤ 10年） | UMS |
| 校验通过 | 状态置为 `under_review`，生成申请编号 | UMS |
| 校验失败 | 状态置为 `auto_check_failed`，显示原因 | UMS |
| 招生审批 | 状态变更为 `accepted`/`rejected`/`waitlisted` | UMS + 通知服务 |

#### 2.1.4 出口标准
| 标准 | 验证方法 |
|------|----------|
| 申请编号生成 | 格式：`APP-2026-XXXX` |
| 状态为 `under_review` 或 `accepted` | 数据库查询确认 |
| 通知已发送 | 检查 Notification 表 |

#### 2.1.5 识别的问题与补充

**问题 1**: 录取通知书自动生成功能缺失
- **现状**: 系统仅发送站内通知，未生成 PDF 格式录取通知书
- **补充方案**: 
  - 添加 `offerLetterAssetId` 字段存储录取通知书
  - 录取时自动调用 PDF 生成服务
  - 通过邮件发送录取通知书附件

**问题 2**: 邮件通知未实际发送
- **现状**: 仅创建 Notification 记录，未调用邮件服务
- **补充方案**: 
  - 集成邮件发送服务（如 Nodemailer）
  - 配置 SMTP 服务器
  - 添加邮件模板支持

**问题 3**: 申请人无法查看已上传文档
- **现状**: 文档上传后无法预览
- **补充方案**: ✅ 已修复 - 添加文档预览功能

### S2: 学籍档案创建

#### 2.2.1 入口条件
| 条件 | 说明 | 验证方法 |
|------|------|----------|
| 申请状态 | `status = 'accepted'` | 数据库查询 |
| 用户身份 | 申请人点击「接受录取」| 前端按钮触发 |
| 学号唯一性 | 学号未分配给其他学生 | 事务锁保证 |

#### 2.2.2 执行流程

```
录取确认流程:
├── 1. 验证申请状态为 'accepted'
├── 2. 检查是否已存在学生记录（幂等性）
├── 3. 事务内生成学号
│   └── 格式: {年份}{序号3位}，如 2026001
├── 4. 创建 Student 记录
│   ├── 绑定 userId
│   ├── 绑定 applicantId
│   ├── 设置 programmeId, intakeId
│   └── 设置奖学金比例（如申请）
├── 5. 更新用户角色为 'student'
└── 6. 发送欢迎通知
```

#### 2.2.3 系统自动执行

| 触发点 | 执行内容 | 集成系统 |
|--------|----------|----------|
| 学号生成 | 使用事务保证序号连续性 | UMS |
| 学生记录创建 | 关联用户、申请、专业、入学期 | UMS |
| 通知发送 | 站内欢迎通知 | 通知服务 |

#### 2.2.4 出口标准
| 标准 | 验证方法 |
|------|----------|
| 学号格式正确 | 匹配正则 `^2026\d{3}$` |
| Student 记录存在 | 数据库查询 `studentId` |
| 用户角色更新 | User.role = 'student' |

#### 2.2.5 识别的问题与补充

**问题 1**: 用户账号自动创建功能缺失
- **现状**: 需要预先创建用户账号
- **补充方案**: 
  - 录取时自动创建用户账号
  - 生成临时密码（格式：`{名字}@{年份}`）
  - 通过邮件发送账号凭据

**问题 2**: 密码重置流程缺失
- **现状**: 新用户无法自助重置密码
- **补充方案**: 
  - 添加「忘记密码」功能
  - 通过邮件发送重置链接

### S3: 选课注册

#### 2.3.1 入口条件
| 条件 | 说明 | 验证方法 |
|------|------|----------|
| 学生身份 | 已完成学籍创建 | User.role = 'student' |
| 学期开放 | 当前学期允许选课 | Semester.isActive = true |
| 学分范围 | 符合学生类型的学分限制 | 前后端双重校验 |

#### 2.3.2 执行流程

```
选课注册流程:
├── 1. 加载可选课程列表
│   ├── 显示课程信息、时间、剩余名额
│   └── 显示先修课程要求
├── 2. 前端实时校验
│   ├── 学分下限（标准学生：12 CH）
│   ├── 学分上限（标准学生：18 CH，CGPA≥3.5：21 CH）
│   └── 时间冲突检测
├── 3. 提交选课请求
├── 4. 后端校验（按顺序）
│   ├── 课程存在性验证
│   ├── 先修课程验证
│   ├── 时间冲突验证
│   └── 学分范围验证
└── 5. 级联操作（事务内）
    ├── 创建 Enrolment 记录
    ├── 生成校园卡号
    ├── 激活图书馆账号
    ├── 激活邮箱账号
    └── 生成费用账单
```

#### 2.3.3 系统自动执行（四系统集成）

| 系统 | 执行内容 | 数据变更 |
|------|----------|----------|
| **LMS** | 创建选课记录 | Enrolment.status = 'registered' |
| **图书馆** | 激活账号 | LibraryAccount.isActive = true |
| **校园卡** | 生成卡号 | Student.campusCardNo = 'CC-2026001' |
| **财务** | 生成账单 | FeeInvoice 记录创建 |

#### 2.3.4 出口标准
| 标准 | 验证方法 |
|------|----------|
| 选课记录创建 | Enrolment 表查询 |
| 校园卡号生成 | Student.campusCardNo 非空 |
| 图书馆账号激活 | LibraryAccount.isActive = true |
| 费用账单生成 | FeeInvoice 表查询 |

#### 2.3.5 识别的问题与补充

**问题 1**: 选课退课后的账单更新
- **现状**: 退课后账单金额未自动调整
- **补充方案**: 
  - 退课时触发账单重新计算
  - 已支付金额转为账户余额

**问题 2**: 选课名额实时性
- **现状**: 名额显示可能有延迟
- **补充方案**: 
  - 使用 WebSocket 实时推送名额变化
  - 添加乐观锁防止超额选课

### S4: 学费缴纳

#### 2.4.1 入口条件
| 条件 | 说明 | 验证方法 |
|------|------|----------|
| 账单生成 | FeeInvoice 记录存在 | 数据库查询 |

#### 2.4.2 执行流程

```
支付流程:
├── 1. 查看账单明细
│   ├── 学费（学分 × 单价）
│   ├── 图书馆费（BND 50 固定）
│   ├── 宿舍押金（可选）
│   └── 奖学金减免
├── 2. 选择支付方式
│   ├── 信用卡/借记卡
│   ├── 网上银行
│   ├── 电子钱包
│   └── QR Pay
├── 3. 模拟支付处理
│   ├── 演示卡号正常：任意卡号
│   └── 演示卡号拒绝：4000 0000 0000 0002
├── 4. 支付成功处理
│   ├── 创建 Payment 记录
│   ├── 更新账单状态
│   └── 发送收据通知
└── 5. 生成/下载收据
```

#### 2.4.3 系统自动执行

| 触发点 | 执行内容 | 集成系统 |
|--------|----------|----------|
| 支付成功 | 更新账单状态 | 财务系统 |
| 金额计算 | outstandingBalance 更新 | 财务系统 |
| 收据生成 | PDF 收据生成 | 文档服务 |
| 通知发送 | 支付成功通知 | 通知服务 |

#### 2.4.4 出口标准
| 标准 | 验证方法 |
|------|----------|
| Payment 记录创建 | transactionRef 非空 |
| 账单状态更新 | status = 'paid' 或 'partial' |
| 收据可下载 | PDF 文件生成成功 |

#### 2.4.5 识别的问题与补充

**问题 1**: 支付网关集成
- **现状**: 仅模拟支付，未接入真实支付网关
- **补充方案**: 
  - 集成 Brunei 本地支付网关
  - 添加支付回调处理
  - 实现支付状态同步

**问题 2**: 分期付款支持
- **现状**: 仅支持全额支付
- **补充方案**: 
  - 添加分期付款计划配置
  - 实现分期还款跟踪

## 六、应急预案

### 6.1 常见故障处理

| 故障现象 | 可能原因 | 处理方法 |
|----------|----------|----------|
| 登录失败 | 数据库连接断开 | 重启后端服务 |
| 页面空白 | 前端编译错误 | 清除缓存重新编译 |
| AI 评分失败 | API Key 无效 | 检查 SystemConfig 配置 |
| 文件上传失败 | 存储空间不足 | 清理临时文件 |
| 支付超时 | 模拟支付服务异常 | 重启后端服务 |

### 6.2 演示中断恢复

```
场景 1: S1 申请过程中断
恢复步骤:
1. 刷新页面
2. 检查 sessionStorage 是否保存表单数据
3. 从上次步骤继续

场景 2: S3 选课过程中断
恢复步骤:
1. 刷新页面
2. 检查已选课程状态
3. 重新选择或确认

场景 3: S4 支付过程中断
恢复步骤:
1. 检查 Payment 记录
2. 确认支付状态
3. 如已扣款但未更新，手动更新账单状态

场景 4: S6 评分过程中断
恢复步骤:
1. 检查 Submission 状态
2. 重新进入评分页面
3. 继续评分操作
```
